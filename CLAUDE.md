# CLAUDE.md

Гайд для работы над проектом. Загружается в контекст каждой сессии — держим коротким и точным.

## Что это

**Property Management System (PMS) + Channel Manager** — SaaS «под ключ» для арендодателей
квартир/апартаментов/отелей. Управление объектами, размещение объявлений на площадках с
автосинхронизацией цен и доступности, единый чат со всех площадок, расчёт комиссий,
заезд/выезд, гостевые страницы с кодами доступа, управление клинингом.

**Стадия:** greenfield, кода ещё нет. Этот файл — конституция проекта, а не описание готового.

## Рынок и интеграции

- Площадки на старте: **Avito, Циан, Суточно, Островок** (РФ).
- **Свои интеграции**, без готовых агрегаторов (Channex и т.п.).
- Channel-manager живёт **внутри монолита**; позже при росте можно вынести в отдельный сервис.

## Стек (жёстко зафиксирован)

- Только **TypeScript / React / JS-экосистема**.
- Backend: **Hono** (чистый API, без SSR).
- Frontend: **Vite + React SPA**, **адаптив** (один UI на все экраны), **без SSR**.
- Server state: **TanStack Query**. Типобезопасность фронт↔бэк: **Hono RPC (`hc`)** + общие zod-контракты.
- Стили: **CSS-in-JS на Emotion** (`@emotion/styled` + `css`-проп), токены через `ThemeProvider`.
- БД: **PostgreSQL через PGlite** (WASM, npm-зависимость, без сервера, persist в папку `.data/`)
  + **Drizzle ORM**. Схема — `apps/api/src/db/schema.ts`; миграции — `apps/api/drizzle/`
  (`pnpm --filter @pms/api db:generate`), применяются на старте. Переезд на серверный Postgres =
  смена драйвера, схема и запросы те же. Репозитории-адаптеры скрывают БД за портами.
- Монорепо: **pnpm workspaces**.
- Мобильное/нативное приложение (WebView) — **отложено**, пока только web.

## Незыблемые принципы

### 1. Функциональный стиль везде
- Только функциональные компоненты + хуки. Никаких классов в домене и use-cases.
- Зависимости — через аргументы функций (composition root на старте), без DI-контейнеров/декораторов.
- Ошибки — `Result`/`Either` (**neverthrow**), а не `throw`. «Functional core, imperative shell»:
  чистая логика в центре, побочки (БД, HTTP, очередь) по краям.

### 2. Никакого глобального стора на фронте
- Запрещены Redux/Zustand/MobX для бизнес-данных. **Источник правды — сервер.**
- «Общее состояние» = кэш TanStack Query. Координация между виджетами — через **инвалидацию
  query-ключей**, не через общий стор.
- **Каждый виджет самодостаточен:** сам вызывает свои query/mutation, сам перезапрашивает.
  Лучше лишний запрос на бэкенд, чем глобальное состояние.
- Локальное UI-состояние (модалки, ввод форм) — `useState`/`useReducer` внутри компонента.
- Допустимые исключения (это не «глобальный стор данных»): **auth** (httpOnly-cookie + тонкий
  `AuthContext`) и **тема/локаль** (через `Context`).
- Реалтайм (инбокс) — WebSocket пишет прямо в кэш через `queryClient.setQueryData`.

### 3. UI только через UI-kit (`shared/ui`)
- Вся вёрстка — из компонентов UI-kit (`Button`, `Input`, `Card`, `Stack`, `Text`, `Heading`,
  `Link`). Импорт из barrel: `import { ... } from 'shared/ui'`.
- **Запрещены инлайн-стили (`style={{…}}`) и произвольный глобальный CSS.** Глобально — только
  сброс в `index.css`. Одноразовые правки раскладки — через `css`-проп Emotion с токенами темы
  (`css={(t) => ({ color: t.colors.danger })}`), не хардкодом.
- Цвета/отступы/радиусы/шрифты — только из `theme` (`shared/ui/theme`). Магических значений нет.
- Новый переиспользуемый элемент — добавляем в `shared/ui` и экспортируем из barrel, а не верстаем
  по месту.

### 4. Компонент `Link` (обязательный паттерн навигации)
- Проп `to` (**обязательный**) — основной адрес. По **чистому левому клику** делаем
  client-routing (`navigate(to)` / `history.push`), с `preventDefault`.
- Проп `nativeHref` (**опциональный**) — то, что реально стоит в `<a href>`; управляет нативными
  действиями браузера (новая вкладка, средняя/правая кнопка). Если не передан — в `href` идёт `to`.
- **Критично:** `preventDefault` только для чистого левого клика. При Ctrl/Cmd/Shift/Alt, не-левой
  кнопке или `target="_blank"` — отдаём управление браузеру (иначе ломается «открыть в новой вкладке»).
- Прямую передачу `href` запрещаем типами (`Omit<…, 'href'>`), чтобы не путать с `to`.

## Архитектура

### Backend — функциональный гексагон (Ports & Adapters)
Зависимости направлены внутрь, к домену. Домен не знает про БД/HTTP/площадки.

```
http (Hono) → application (use-cases) → domain (pure)
                     ↓                       ↑
                  ports (интерфейсы) ← adapters (Postgres, каналы, очередь)
```

- `domain/` — типы + pure-функции правил, `Result`. Без IO.
- `ports/` — интерфейсы: `ReservationRepo`, `ChannelAdapter`, `Clock`, `IdGen`…
- `application/` — use-cases: оркестрируют домен + порты, возвращают `Result`.
- `adapters/` — реализации: Postgres-репо, **адаптеры площадок**, HTTP-клиенты, очередь (BullMQ).
- `http/` — тонкие роуты: zod-parse → use-case → map `Result` в HTTP.

**Channel-manager:** порт `ChannelAdapter`; каждая площадка (Avito/Циан/Суточно/Островок) —
отдельный адаптер-функция. Способ связи (API/iCal/парсинг) спрятан внутри адаптера.
Синхронизация через очередь, обязательно **идемпотентно** (защита от двойных броней/апдейтов).

### Frontend — функциональный Feature-Sliced Design
Зависимости текут только вверх: `shared → entities → features → widgets → pages → app`.

- `shared/` — api-клиент (Hono RPC), ui-kit (`Link`, `Button`…), хелперы, конфиг.
- `entities/` — доменные модели + query-хуки чтения (`useReservation`, типы, мапперы).
- `features/` — действия пользователя = mutation-хуки + мелкий UI (`changePrice`, `sendMessage`).
- `widgets/` — самодостаточные блоки, владеют своими данными (Calendar, InboxThread, PricePanel).
- `pages/` — компонуют виджеты, роутинг.
- `app/` — провайдеры (QueryClient, Router, ErrorBoundary), точка входа.

## Базовые модули домена

1. **Identity & Tenancy** — организации, пользователи, роли (owner/manager/cleaner/guest), auth, инвайты.
2. **Properties** — объекты, атрибуты, фото, контент под каждый канал.
3. **Channels** — подключение площадок, маппинг «объект ↔ листинг», адаптеры, движок синхронизации.
4. **Calendar & Availability** — доступность, блокировки, защита от overbooking.
5. **Reservations** — жизненный цикл брони, источник-канал, даты заезда/выезда, статусы.
6. **Pricing** — базовая цена, правила, наценки/комиссии по каналам, проброс цены в каналы.
7. **Commissions & Billing** — настройка комиссий per-channel, полуавтоматический расчёт, отчёты.
8. **Messaging / Unified Inbox** — треды со всех каналов, отправка/приём, реалтайм.
9. **Guest Portal** — публичная страница гостя: коды доступа, инструкции заезда.
10. **Operations / Cleaning** — задачи уборки от выездов, назначение, статусы, экран клинера.
11. **Notifications** — email / push / in-app.

Сквозное: audit log, очередь задач, идемпотентность, деньги в minor units (целые копейки, без float).
Система **multi-tenant** (SaaS) с первого дня.

## Подписки и триал (SaaS-биллинг тенанта)

Это про оплату **нами** (org платит за PMS), не про комиссии по броням (модуль 7). Живёт в
`modules/subscriptions` (домен/порты) + `modules/payments` (исполнение платежей, gateway).

- **Cardless-триал.** Стратегия: высокая конверсия на входе + дорогой обход, а не хард-блоки.
  Длина триала настраивается на плане (`Plan.trialDays`, типично 2–4 нед., дефолт 14).
- **Вход решает `decideTrialPolicy`** (pure, [[trial-policy]]):
  нет подтв. телефона → `reject`; номер уже жёг триал ИЛИ high-risk → `require_card_first`; иначе `grant_trial`.
  Телефон подтверждаем **звонком** (предпочтительно). Один номер = один cardless-триал навсегда
  (`TrialEligibilityLedger`, cross-tenant, нормализация E.164).
- **`require_card_first`** = auth-hold (₽1–₽10 → void, не charge+refund) + триал остаётся бесплатным,
  но **автоконвертируется** в платный в конце; отпечаток карты → в ledger (одна карта = один триал).
- **Мягкие сигналы** (IP/fingerprint/email-домен) → `RiskScorer` → только усиление трения, не хард-блок;
  всё пишем в audit log для пост-анализа паттернов.
- **Машина статусов** (`subscriptions/domain/subscription.ts`, pure, `Result`):
  `trialing → active` (оплата/автобиллинг) · `trialing → expired` (конец без оплаты) ·
  `expired → active` (оплата) · `active → canceled`. По истечении неоплаченного триала —
  **read-only НАВСЕГДА до оплаты** (`expired`), данные не удаляем. `isReadOnly` гейтит запись,
  `isTrialUnpaid` (`!everPaid`) питает DetachGate.
- **Анти-абьюз привязки аккаунтов площадок** ([[account-binding]]): при подключении площадки
  фиксируем `(platform, externalAccountId) → orgId` в глобальном реестре (whoami через
  `AccountIdentity.identify`). Чужой подтверждённый аккаунт увести нельзя (`conflict`); пока триал
  не оплачен — отвязать нельзя (`decideDetach`, замок от `isTrialUnpaid`). Итог: повтор триала с
  каналами требует НОВЫХ реального телефона + реального аккаунта площадки — невыгодно.

## Интеграции с площадками (channel-manager)

- **Официальный API/фид — стратегия по умолчанию.**
  - *Avito:* XML-автозагрузка (контент, тариф Avito Pro) + REST API (`api.avito.ru`, OAuth2 Bearer):
    цены по датам `POST /realty/v1/.../prices`, календарь занятости `POST /core/v1/.../bookings`,
    брони `GET /realty/v1/.../bookings`, мессенджер (вебхуки, подпись `x-avito-messenger-signature`,
    ответ ≤2 сек).
  - *Cian:* публикация — XML-фид (формат Циан 2.0), pull раз в несколько часов; цена меняется
    перевыкладкой фида. Всё остальное — REST API `public-api.cian.ru` (Bearer `<ACCESS KEY>` на
    аккаунт, **не** OAuth2; ключ выдаётся вручную через import@cian.ru): сообщения и чаты
    (вебхуки v3 push, без подписи; `send-message`), обратная связь по публикации
    (`get-order`/`get-my-offers` — статусы и пообъектные ошибки), модерация неактуальности
    (`get-stale-offers` + confirm/reject), статистика, биллинг, аукцион. Лиды приходят звонками
    (`get-calls-report`), брони/резерваций у Циана нет. Ответы обёрнуты в `{ operationId, result }`,
    списки — повтором параметра (`?offerIds=1&offerIds=2`), лимит ≤10 req/sec на метод.
- **Скрейпинг/эмуляция пользователя запрещены как основная стратегия.** Допустим лишь как
  изолированный адаптер последней инстанции за тем же портом и только для критичной возможности,
  у которой нет официального API. Причина: баны бьют по аккаунтам клиентов (это SaaS).
- **Capability-based порты.** Адаптер декларирует `ChannelCapabilities` и реализует подмножество
  портов (`ListingPublisher`, `PriceSync`, `AvailabilitySync`, `BookingSource`, `MessagingChannel`,
  `WebhookParser`). Возможности у площадок неравные — система деградирует мягко.
- **Унификация push/poll — на уровне нормализованного стора, не в сигнатурах адаптера.** Приём
  входящих (сообщения/брони) описывается `IngestionStrategy<E>` = `push | poll | none`. Единый
  Ingestion Runner сам решает — регистрировать вебхук или крутить поллинг — и сводит всё в
  нормализованный стор (идемпотентно по `externalId`). Потребитель (use-case/виджет) читает всегда
  одинаково и не знает ни про площадку, ни про push/poll.
- **Фронт никогда не ходит на площадки** — только на наш бэкенд (TanStack Query) + наш realtime
  (SSE/WS → `queryClient.setQueryData`). Получили данные вебхуком или поллингом — фронту без разницы.
- Деньги — minor units; в фид цена конвертируется в рубли. Идемпотентность обязательна и для
  outbox (исходящие), и для inbox (входящие).
